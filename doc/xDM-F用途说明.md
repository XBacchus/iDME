# 基于 xDM-F 的核心工作说明

## 1. 我使用 xDM-F 做了什么

在本项目中，xDM-F 不是“存数据的附属工具”，而是业务数据层的核心引擎。我主要用它完成了以下工作：

1. 设计并发布工业业务数据模型  
   - 设备模型：`Equipment`  
   - 物料模型：`Part`  
   - 工序模型：`WorkingProcedure`  
   - 工艺模型：`WorkingPlan`

2. 建立模型之间的业务关系  
   - 工艺 与 工序 的关系  
   - 工序 与 设备 的关系  
   - 工序 与 物料 的关系  
   通过关系实体/参考对象表达“工艺编排、设备执行、物料消耗”的真实工业关联。

3. 承载并约束核心业务字段  
   - 设备：编码、名称、厂家、品牌、规格、供应商、生产日期、使用年限、折旧方式、位置等  
   - 物料：编号、名称、规格、库存、供应商，支持分类和版本  
   - 工序：编号、名称、步骤、设备、操作人、开始时间、结束时间  
   - 工艺：编号、名称、版本、操作信息、设备使用情况等

4. 通过运行态 API 对外提供统一数据服务  
   模型完成后，下载并部署 xDM-F 运行态 SDK，业务层统一调用 xDM-F API 完成新增、查询、修改、删除及关系维护，保证前后端与数据层交互一致。

---

## 2. xDM-F 在本项目中的价值

1. 统一数据语义  
   将“设备-物料-工序-工艺”抽象为标准模型，避免前后端各自定义口径。

2. 模型驱动开发  
   先建模、再开发，减少重复建表和字段返工，提高功能迭代效率。

3. 支撑复杂工业关系  
   用关系建模表达 BOM、工艺路线和工序执行链路，便于查询与追溯。

4. 满足赛项硬性要求  
   业务层、展示层与数据层交互全部通过 xDM-F API，避免绕开 xDM-F 带来的失分风险。

---

## 3. xDM-F 直接支撑的功能模块

1. 设备管理  
   基于 `Equipment` 模型实现设备档案的增删改查与扩展属性管理。

2. 物料与 BOM 管理  
   基于 `Part` 模型实现物料维护、分类管理、版本管理、编码/名称搜索与 BOM 关系构建。

3. 工序配置  
   基于 `WorkingProcedure` 完成 5 道标准工序配置：毛坯制造、粗加工、精加工、检测、入库。

4. 工艺路线配置  
   基于 `WorkingPlan` 配置工艺路线“中心轮零件加工（V1.0）”，并正确关联工序顺序。
