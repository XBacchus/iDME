# Day1 需求文档：建模与环境打底

## 1. 当日目标

完成“可开发状态”准备：

1. xDM-F 四个核心模型与四条核心关系创建并发布。
2. 运行态 SDK 启动可用，健康检查通过。
3. 前后端本地开发环境可启动。
4. 固化模型截图、关系截图、配置基线。

## 2. 并行任务包

### 任务包 A（xDM-F 组）

输入：

1. 赛题模型字段要求。
2. 固定工序与工艺路线要求。

输出：

1. 已发布模型：`Equipment`、`Part`、`WorkingProcedure`、`WorkingPlan`。
2. 已发布关系：`WorkingPlan -> WorkingProcedure`、`WorkingProcedure -> Equipment`、`WorkingProcedure -> Part`、`Part -> Part`。
3. 证据截图：模型列表、关系列表、图谱页面。

验收标准：

1. 模型和关系状态均为“已发布”。
2. 英文模型名与要求一致。

### 任务包 B（运行态部署组）

输入：

1. SDK 包与 license 文件。
2. 数据库与 Redis 连接信息。

输出：

1. 运行态服务成功启动。
2. 健康检查接口返回成功。
3. 本地登录鉴权可进入控制台（LOCAL 模式）。

验收标准：

1. `GET /v1/health` 返回成功。
2. `index.html` 可进入登录页并登录。

### 任务包 C（后端基础组）

输入：

1. `miniapp/backend` 现有工程。

输出：

1. 启动配置确认：JDK17、端口、基础配置。
2. xDM-F Client 基础封装骨架（不写具体业务）。
3. 接口目录与模块边界定义（设备/物料/工序/工艺）。

验收标准：

1. 后端可本地启动。
2. 模块边界明确，避免后续散落调用。

### 任务包 D（前端基础组）

输入：

1. `miniapp/frontend` 现有工程。

输出：

1. 路由与页面骨架：设备、物料、工序、工艺路线。
2. API 调用统一走 `/api` 代理。
3. 基础状态处理：加载态、空态、错误提示占位。

验收标准：

1. 前端可本地启动。
2. 菜单可访问四个页面骨架。

## 3. 串行依赖

1. 任务包 A 完成后，Day2 后端业务接口才能稳定联调。
2. 任务包 B 完成后，Day2 开始真实 API 调用验证。

## 4. 当日产物清单

1. 模型与关系发布截图。
2. 运行态健康检查记录。
3. 前后端启动截图或日志片段。
4. Day1 完成状态表（完成/风险/阻塞）。
